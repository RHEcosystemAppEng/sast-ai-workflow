## Deploying the SAST AI Workflow

This guide covers deployment on a local OpenShift cluster using CodeReady Containers (CRC) or an existing OpenShift cluster.

### 1. Install CRC (Local Development)

**For existing OpenShift clusters, skip to step 2.**

1. Download CRC from: https://developers.redhat.com/products/openshift-local/overview
2. Install and setup:
   ```bash
   crc setup
   crc config set disk-size 100  # Minimum 60GB recommended
   crc start
   crc status  # Verify installation
   ```

### 2. Install OpenShift Pipelines Operator

Install the OpenShift Pipelines operator from OperatorHub in the OpenShift console.

```bash
oc whoami --show-console  # Get console URL
```

### 2.5. Understanding the Kustomize Architecture

This deployment uses **Kustomize overlays** for environment-specific configurations:

```
deploy/tekton/
├── base/                      # Base Tekton resources (common to all environments)
│   ├── pipeline.yaml
│   ├── tasks/
│   └── configmaps/
└── overlays/
    ├── dev/                   # Development overlay (:latest image, Google Drive)
    │   ├── kustomization.yaml
    │   └── service-accounts/  # Generated by prepare-kustomize.sh
    ├── prod/                  # Production overlay (versioned image, Google Drive)
    │   ├── kustomization.yaml
    │   └── service-accounts/  # Generated by prepare-kustomize.sh
    └── mlops/                 # MLOps overlay (S3/MinIO, DVC)
        ├── kustomization.yaml
        └── service-accounts/  # Generated by prepare-kustomize.sh
```

**Deployment Flow:**
1. You configure `.env` file with credentials and settings
2. `prepare-kustomize.sh` reads `.env` and generates environment-specific files
3. Kustomize builds final resources by merging base + overlay + generated files
4. Resources are applied to the cluster

**Key Benefits:**
- ✅ Secrets never committed to Git (generated from `.env`)
- ✅ One base configuration, multiple environment overlays
- ✅ GitOps-friendly (ArgoCD can sync from Git)
- ✅ Clear separation between dev/prod/mlops environments

### 3. Setup Environment and Secrets

#### 3.1. Create .env File

Create a `.env` file in the project root:

```env
# Required: GitLab credentials
GITLAB_TOKEN=your_gitlab_token_here

# Required: LLM Configuration  
LLM_URL=https://your-llm-endpoint.com/v1
LLM_API_KEY=your_llm_api_key_here
LLM_API_TYPE=openai
LLM_MODEL_NAME=your-llm-model-name

# Required: Embeddings LLM Configuration
EMBEDDINGS_LLM_URL=https://your-embeddings-endpoint.com/v1
EMBEDDINGS_LLM_API_KEY=your_embeddings_api_key_here
EMBEDDINGS_LLM_MODEL_NAME=your-embeddings-model-name

# Required: Google Service Account (for Drive/Sheets access)
GOOGLE_SERVICE_ACCOUNT_JSON_PATH=./service_account.json

# Optional: GCS Service Account (for uploading SARIF to GCS bucket)
GCS_SERVICE_ACCOUNT_JSON_PATH=./gcs_service_account.json

# Optional: S3/MinIO Configuration (for MLOps environment)
AWS_ACCESS_KEY_ID=your_s3_access_key_id
AWS_SECRET_ACCESS_KEY=your_s3_secret_access_key
S3_ENDPOINT_URL=https://your-minio-endpoint.com
S3_INPUT_BUCKET_NAME=mlops-input
S3_OUTPUT_BUCKET_NAME=mlops-output

# Optional: DVC Configuration (for MLOps environment)
DVC_REPO_URL=https://github.com/your-org/sast-ai-dvc
DVC_DATA_VERSION=v2.0
```

#### 3.2. Prepare Prerequisites

1. Place your Google service account JSON file (for Drive/Sheets) in project root
2. (Optional) Place your GCS service account JSON file if using GCS bucket upload
3. Login to container registry: `podman login quay.io` (or `docker login quay.io`)

#### 3.3. Deployment Workflow

The new Kustomize-based deployment uses a two-step process:

**Step 1: Prepare secrets and service accounts**
```bash
cd deploy
./scripts/prepare-kustomize.sh dev   # or prod, mlops
```

This script:
- Loads variables from `.env` file
- Creates secret environment files (`secrets-*.env`)
- Copies service account JSON files to overlay directory
- Validates all required configuration

**Step 2: Deploy with Kustomize**
```bash
make deploy-dev     # Development environment
make deploy-prod    # Production environment (requires IMAGE_VERSION)
make deploy-mlops   # MLOps environment (S3/MinIO + DVC)
```

The Makefile automatically runs `prepare-kustomize.sh` and then applies Kustomize overlays.

### 4. Resource Naming Convention

All SAST AI Workflow resources use the `sast-ai-` prefix for easy identification and management:

**Secrets:**
- `sast-ai-gitlab-token` - GitLab access token
- `sast-ai-default-llm-creds` - LLM and embeddings API credentials
- `sast-ai-google-service-account` - Google service account JSON for spreadsheet access
- `sast-ai-gcs-service-account` - GCS service account JSON for uploading SARIF reports to GCS bucket (optional)
- `sast-ai-s3-output-credentials` - S3/Minio output access credentials for MLOps environment (optional)
- `sast-ai-quay-registry-config` - Container registry pull credentials

**ConfigMaps:**
- `sast-ai-upload-to-drive` - Google Drive upload scripts
- `sast-ai-upload-to-gcs` - GCS bucket upload scripts for SARIF reports
- `sast-ai-upload-to-s3-output` - S3/MinIO output upload scripts (for mlops overlay)

This naming convention allows administrators to quickly filter and manage SAST AI resources:
```bash
# View all SAST AI resources
oc get all,secrets,configmaps -l app=sast-ai
# Or use grep filtering
oc get secrets | grep sast-ai
```

### 5. Makefile Commands

| Command | Description |
|---------|-------------|
| **Main Deployment** | |
| `make deploy-dev` | Deploy development environment (:latest tag, dev overlay) |
| `make deploy-prod IMAGE_VERSION=x.y.z` | Deploy production environment (versioned tag, prod overlay) |
| `make deploy-mlops` | Deploy MLOps environment (S3/MinIO, DVC, mlops overlay) |
| **Execution** | |
| `make run` | Execute pipeline with parameters from .env or command line |
| **Cleanup** | |
| `make clean` | **⚠️ Deletes ALL resources in namespace** (validates critical namespaces) |
| **Internal Targets** | |
| `make prepare` | Run prepare-kustomize.sh script (automatically called by deploy-*) |
| `make argocd-setup-project` | Setup ArgoCD AppProject (automatically called by deploy-*) |

**Examples:**
```bash
# Development deployment
make deploy-dev

# Production deployment with specific version
make deploy-prod IMAGE_VERSION=1.2.3

# MLOps deployment
make deploy-mlops

# Run pipeline with custom parameters
make run PROJECT_NAME=myapp REPO_REMOTE_URL=https://github.com/user/repo

# Clean up everything
make clean ENV=dev
```

### 6. Quick Start

#### 6.1. Create OpenShift Project

```bash
oc new-project sast-ai-workflow
oc project sast-ai-workflow
```

#### 6.2. Deploy Development Environment

```bash
cd deploy
make deploy-dev
```

This will:
1. Run `prepare-kustomize.sh dev` to generate secrets and copy service accounts
2. Deploy Tekton resources via `oc apply -k tekton/overlays/dev`
3. Setup ArgoCD Application for GitOps sync

#### 6.3. Deploy Production Environment

```bash
cd deploy
make deploy-prod IMAGE_VERSION=1.2.3
```

Production deployment requires a specific image version (not `:latest`).

#### 6.4. Deploy MLOps Environment (S3/MinIO + DVC)

```bash
cd deploy
make deploy-mlops
```

MLOps environment requires additional `.env` variables:
- `DVC_REPO_URL`, `DVC_DATA_VERSION`
- `S3_ENDPOINT_URL`, `S3_INPUT_BUCKET_NAME`, `S3_OUTPUT_BUCKET_NAME`
- `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`

#### 6.5. Run the Pipeline

```bash
make run PROJECT_NAME=myapp REPO_REMOTE_URL=https://github.com/user/repo
```

Or customize with additional parameters:
```bash
make run \
  PROJECT_NAME=systemd \
  REPO_REMOTE_URL=https://github.com/systemd/systemd \
  INPUT_REPORT_FILE_PATH=https://docs.google.com/spreadsheets/d/YOUR_SHEET
```

### 7. Manual Deployment (Advanced)

If you prefer to run Kustomize commands directly without the Makefile:

```bash
# Step 1: Prepare secrets and service accounts
cd deploy
./scripts/prepare-kustomize.sh dev   # or prod, mlops

# Step 2: Apply Kustomize overlay
oc apply -k tekton/overlays/dev      # or prod, mlops

# Step 3: (Optional) Setup ArgoCD for GitOps
oc apply -f argocd/appproject.yaml
oc apply -f argocd/argocd-application-dev.yaml
```

**Recommended:** Use `make deploy-dev/prod/mlops` instead for simplified deployment.

### 8. GitOps with ArgoCD

For VPN-protected clusters, use GitOps to automatically sync Tekton resources from GitHub.

#### 8.1. Prerequisites
- ArgoCD installed in a **dedicated management namespace** (recommended: `openshift-gitops` or `argocd`)
- Repository access from within the cluster

**⚠️ Best Practice:** Keep ArgoCD in a separate namespace from your managed applications for:
- Security isolation
- Clear separation of concerns
- Multi-tenancy support
- Avoiding circular dependencies

#### 8.2. Deploy with GitOps

**ArgoCD integration is automatic** - when you run `make deploy-dev/prod/mlops`, the Makefile:
1. Creates ArgoCD **AppProject** named `sast-ai` (if not exists)
2. Labels the target namespace for ArgoCD management
3. Deploys ArgoCD Application to watch Git and sync to cluster

The AppProject provides:
- 🔒 **Security**: Only allowed namespaces and repos
- 🎯 **Resource Control**: Only Tekton resources permitted
- 🔍 **Audit Trail**: Clear project boundaries

**Deploy with custom ArgoCD namespace:**
```bash
# Default ArgoCD namespace is openshift-gitops
make deploy-dev

# Custom ArgoCD namespace
make deploy-dev ARGOCD_NAMESPACE=my-argocd-namespace

# Different target namespace
make deploy-dev NAMESPACE=sast-ai-dev
```

**What Gets Deployed:**
1. ArgoCD AppProject `sast-ai` (if not exists) - defines security boundaries
2. Target namespace labeled with `argocd.argoproj.io/managed-by=<ARGOCD_NAMESPACE>` - enables ArgoCD management
3. ArgoCD Application - watches Git repository and syncs Tekton resources to cluster

**AppProject Security Features:**
- ✅ Allowed destinations: Dynamically added when you deploy (any namespace you specify)
- ✅ Allowed resources: Tekton Pipelines, Tasks, ConfigMaps, Secrets only
- ❌ Blocked resources: LimitRange, ResourceQuota (prevent quota tampering)
- ✅ Source repos: Only your configured Git repository

**Automated Setup:**
- Namespace automatically added to AppProject destinations when you deploy
- Namespace labeled for ArgoCD management
- ArgoCD creates Role + RoleBinding automatically
- No manual permission setup required!

**Common Error**: If you see "namespace X is not managed":
1. Check the namespace label: `oc get namespace <NS> -o jsonpath='{.metadata.labels}'`
2. Verify AppProject destinations: `oc get appproject sast-ai -n <ARGOCD_NS> -o jsonpath='{.spec.destinations[*].namespace}'`
3. Try rerunning the deploy command - it will add the namespace automatically

#### 8.3. Environment Differences

| Environment | Kustomize Path | Use Case | Image Strategy |
|-------------|----------------|----------|----------------|
| **dev** | `deploy/tekton/overlays/dev` | Development/testing | Latest tag, auto-update |
| **mlops** | `deploy/tekton/overlays/mlops` | MLOps with S3 storage | Latest tag, auto-update |
| **prod** | `deploy/tekton/overlays/prod` | Production | Versioned, manual update |

#### 8.4. How It Works
- **Auto-sync**: Changes to `main` branch deploy automatically (~3 min)
- **Self-healing**: Manual changes are automatically reverted
- **Pruning**: Deleted files are removed from cluster
- **Path**: Syncs the appropriate directory based on environment

#### 8.5. Configuration (Optional)
Set in `.env` file:
```env
GITHUB_REPO_URL=https://github.com/your-org/sast-ai-workflow.git
```

#### 8.6. Configuration Changes with GitOps

When you push changes to the Git repository:

**Automatic Sync:**
- Changes to Tekton Pipelines, Tasks, ConfigMaps are automatically synced by ArgoCD
- Sync happens within ~3 minutes of pushing to the `main` branch

**Secrets and Service Accounts:**
- Secrets are generated by Kustomize `secretGenerator` from files created by `prepare-kustomize.sh`
- These files are NOT in Git (they're in `.gitignore`)
- To update secrets: update `.env` file, re-run `make deploy-<env>`

**Prompts:**
- Prompts are bundled inside the container image (not in ConfigMaps)
- To update prompts: rebuild the container image with new prompt files

### 9. Optional S3 Output Upload Configuration

The SAST AI Workflow supports optional S3/Minio output upload for the final Excel analysis results.

#### 9.1. How It Works

The pipeline has two deployment options:

1. **Base Pipeline** - No output upload functionality
2. **MLOps Overlay** - Adds S3/Minio output upload capability

The base pipeline (`deploy/tekton/base/`) contains no storage-specific logic. The mlops overlay (`deploy/tekton/overlays/mlops/`) uses JSON 6902 patches to inject the S3 output upload parameter and step:

```
deploy/tekton/overlays/mlops/
├── kustomization.yaml           # Overlay configuration
├── pipeline-params-patch.yaml   # Adds S3 output parameters (S3_OUTPUT_BUCKET_NAME)
└── task-s3-output-patch.yaml    # Replaces upload-to-gdrive with upload-to-s3-output step
```

#### 9.2. S3/Minio Configuration

To use S3/Minio storage:
```env
# Required
AWS_ACCESS_KEY_ID=your_access_key
AWS_SECRET_ACCESS_KEY=your_secret_key
S3_OUTPUT_BUCKET_NAME=your-bucket-name
S3_ENDPOINT_URL=https://minio.example.com
```

**Notes:**
- `S3_ENDPOINT_URL` is only needed for Minio or non-AWS S3-compatible storage
- For AWS S3, leave `S3_ENDPOINT_URL` empty
- Secrets are automatically generated by `prepare-kustomize.sh` when you run `make deploy-mlops`

#### 9.3. File Organization in S3

Files uploaded to S3 follow this directory and naming pattern:
```
{pipeline-run-id}/{repo-name}_sast_ai_output.xlsx
```

Example:
```
87270e71-8fcf-4d1c-9ae0-b50299df5112/systemd_sast_ai_output.xlsx
```

Where:
- `pipeline-run-id`: The unique Tekton PipelineRun UID (automatically injected via `$(context.pipelineRun.uid)`)
- `repo-name`: The project name (from `PROJECT_NAME` parameter)
- The pipeline run ID is automatically provided by Tekton, ensuring each run has a unique S3 directory

### 10. Parallel Investigation Processing

The workflow processes up to **5 issues concurrently** by default (configurable via `MAX_CONCURRENT_INVESTIGATIONS`).

**Performance:** Sequential (1 worker) = ~6.7 hours for 100 issues. Parallel (5 workers) = ~1.3 hours (5x faster).

**Configuration:**
```yaml
# In PipelineRun:
params:
  - name: MAX_CONCURRENT_INVESTIGATIONS
    value: "10"  # 1 = sequential, 5 = default, 10 = high throughput
```

**Resource Requirements:**

| Workers | Memory | CPU | Use Case |
|---------|--------|-----|----------|
| 1 | 2-3 GB | 1-2 cores | Small reports, debugging |
| 5 (default) | 4-6 GB | 2-4 cores | Most workloads |
| 10 | 6-8 GB | 3-5 cores | Large reports (200+ issues) |

Current allocation: 4 GB request / 6 GB limit, 2-4 CPU cores (supports up to 5-7 workers).

**Logs:** Processing mode indicated by `[sequential]` or `[parallel]` prefix in logs.

### 11. Troubleshooting

#### General Issues
- **View logs:** `oc logs -l tekton.dev/pipelineRun=sast-ai-workflow-pipelinerun -f`
- **Clean environment:** `make clean` (⚠️ deletes everything)
- **Check secrets:** `oc get secrets`
- **Manual pipeline execution:** Use `make run` or execute via OpenShift console

#### S3/Minio Issues
- **Check S3 credentials secret exists:** `oc get secret sast-ai-s3-credentials`
- **Verify secret contents:** `oc get secret sast-ai-s3-credentials -o yaml`
- **Test S3 connectivity:** Check upload-to-s3 step logs in pipeline run
- **Common issues:**
  - Missing endpoint URL for Minio (add `S3_ENDPOINT_URL` to `.env`)
  - Incorrect bucket permissions (ensure write access to bucket)
  - Network connectivity to S3/Minio endpoint
